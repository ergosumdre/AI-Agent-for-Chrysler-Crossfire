Introduction

Large Language Models (LLMs), like ChatGPT and Llama 3, have become incredibly popular for their ability to understand and work with complex information. However, these general-purpose LLMs aren't specifically trained on niche topics or specialized data (Ling, et al., 2024). This means they lack the specific knowledge and understanding needed to deal with the nuances of a particular subject, like the Chrysler Crossfire vehicle.

Online communities dedicated to specific vehicles, like the Chrysler Crossfire, are rich sources of information. Owners share tips, tricks, troubleshooting advice, and repair procedures. However, this knowledge is often spread across forums, comments, and other online platforms, making it hard to find and understand. Furthermore, Crossfire communities often have their own specialized language, common issues, and unwritten rules that a general LLM wouldn't grasp.

This project explores the idea of fine-tuning an LLM using the wealth of community-generated content about the Chrysler Crossfire vehicle. The goal is to create a model that understands the specific challenges and solutions related to this vehicle. By fine-tuning a LLM on Chrysler Crossfire-specific data, it can learn the common jargon, original manufacturer equipment (OEM) part numbers, identify recurring problems, and ultimately facilitate troubleshooting, diagnostics, and repair procedures for individuals, even those with limited automotive expertise. The aim is to bridge the knowledge gap and empower Chrysler Crossfire owners to better understand and maintain their vehicles.

  • Deepseek's API to run Augmented Toolkit in order to generate synthetic data

Methods

This project attempts to develop a large language model that understands the specific challenges and solutions related to the Chrysler Crossfire vehicle. Our approach involved several key stages: data collection and preparation, conversational dataset generation using AugmentToolKit, utilizing Meta's open source Llama 3 Herd of Models, then fine-tuning the model using Unsloth, as shown in Figure 1.

clean_forum_post = function(text) {
  text = str_replace_all(text, "\\{\"@context\".*?\\}", "")
  text = str_replace_all(text, "vbmenu_register\\(.*?\\);", "")
  text = str_replace_all(text, "Share Options", "")
  text = str_replace_all(text, "#\\d+\\s*\\(permalink\\)", "")
  text = str_replace_all(text, "Joined:.*?Likes:", "")
  text = str_replace_all(text, "Posts:.*?From:", "")
  text = str_replace_all(text, "Quote:\\s*Originally Posted by.*?:", "")
  text = str_replace_all(text, "\\w{3} \\d{1,2}, \\d{4} \\| \\d{2}:\\d{2} [AP]M", "")
  text = str_squish(text)
  text = str_trim(text)
  text = ifelse(text == "" | text == "0", NA, text)
  return(text)
}

# Function to parse HTML files from subdirectories
parse_forum_posts = function(base_directory) {
  # List of subdirectories to process
  subdirectories = c(
    'all-crossfires', 
    'crossfire-coupe', 
    'crossfire-roadster', 
    'crossfire-srt6', 
    'troubleshooting-technical-questions-modifications', 
    'tsbs-how-articles', 
    'wheels-brakes-tires-suspension'
  )
  
  forum_posts_list = list()
  
  for (subdir in subdirectories) {
    full_path = file.path(base_directory, subdir)
    
    html_files = list.files(path = full_path, pattern = "\\.html$", full.names = TRUE)
    html_files = html_files[!grepl("/index\\.html$", html_files, ignore.case = TRUE)]
    
    # Process each HTML file
    for (file in html_files) {
      tryCatch({
        html_content = read_html(file)
        posts = html_content %>%
          html_nodes("div.tpost") %>%  # Target content div
          html_text(trim = TRUE) %>% 
          str_replace_all("\\s+", " ") %>%     # Normalize whitespace
          str_replace_all("[\r\n]+", "\n")     # Standardize line breaks
        
        cleaned_posts = sapply(posts, clean_forum_post)
        cleaned_posts = cleaned_posts[!is.na(cleaned_posts)]
        combined_posts = paste(cleaned_posts, collapse = " || ")
        webpage_entry = data.frame(
          subdirectory = subdir,
          filename = basename(file),
          post_content = combined_posts,
          stringsAsFactors = FALSE
        )
        
        forum_posts_list[[file]] = webpage_entry
      }, error = function(e) {
        cat("Error processing file:", file, "\n")
        cat("Error message:", conditionMessage(e), "\n")
      })
    }
  }
  forum_posts_df = do.call(rbind, forum_posts_list)
  return(forum_posts_df)
}

Augmented Data Generation

AugmentToolKit (ATK)

AugmentToolKit is an open-source toolkit designed to automatically convert raw text into high-quality, custom datasets for training and/or fine-tuning LLMs (Armstrong et al, 2024). ATK offers several data-generation pipelines to build datasets with little to no manual intervention. In our workflow, ATK first analyzes an input folder, then converts all files in the input folder into single text file, including forum posts, and PDF files. It then uses a user-specified LLM to generate questions for each line of text, followed by a series of data integrity checks to ensure quality, as shown in Figure 2.

Augmentoolkit 2.0, 2024


For this project, we use the multi-turn conversational question and answer pipeline to produce synthetic data for fine-tuning. Here we utilized Deepseek’s API for the LLM due to its absence of rate limits and low token count pricing. This process ultimately generated 8,385 conversational question and answer pairs that we then used to fine-tune our model.

The Llama 3 Herd of Models


In mid-2024, Meta introduced a herd of Large Language Models (LLMs) called Llama 3 designed to compete with current state-of-the-art (SORA) models, including Google’s Gemma 2, Mistral’s Mixtral, OpenAI’s GPT-4/GPT-4o, Nvidia’s Nemotron, and Anthropic’s Claude 3.5 Sonnet. This herd of models, all built on transformer architecture, came in several flavors to suit different computational needs — from lightweight 8B versions optimized for edge devices, to high-capacity 70B/405B variants designed for coding, reasoning, multilingual, and tool-based tasks (Meta, 2024). These each of the 8B, 70B, 405B parameters models was release as a multilingual Base or Instruct model.

Base Model vs. Instruct Model

In terms of design and applications, the Llama Base Model was trained on a large amount of unlabeled data to understand language and perform generation tasks. This format is great for open-ended tasks like research, creative writing, or general-purpose NLP but requires additional fine-tuning for specialized applications.

On the other hand, the Llama Instruct Model is a fine-tuned version of the base model, optimized for following user instructions and executing specific tasks. It is pre-trained using supervised fine-tuning (SFT) and reinforcement learning from human feedback (RLHF), making it highly reliable for task-oriented applications like chatbots.

For this project, we fine-tuned Meta’s Llama 3.1 8B Instruct Model using Unsloth, a framework dedicated to fine-tuning LLMs.

The first step in developing our model involved curating a dataset that comprises Chrysler Crossfire-related conversations. Oftentimes, these Chrysler Crossfire-related conversations are in the form of user-generated online content, such as online forums. These forums are typically a community-driven online resource where users can find information about their vehicles, such as ‘do it yourself repairs’, ‘common issues’, troubleshooting guides, aftermarket modifications, performance tuning tips, buying and selling advice, and general discourse about ownership experiences. Modeling and Results

Fine-tuning the model

Unsloth

Unsloth is a framework designed to streamline the entire training workflow, from data preparation and loading/quantizing various models to training, evaluating, running, saving, and exporting the model. A key part of preparing data involves the tokenizer, which converts human-readable text into numerical tokens the model can understand. To begin the process, Unsloth first loads in a quantized model, in our case the Llama 3.1 8B 4bit model, along with its associated tokenizer.

With the model, tokenizer, and dataset prepared, we configure the SFTTrainer. This involves setting hyperparameters within TrainingArguments. We define the per_device_train_batch_size (set to 2) and gradient_accumulation_steps (set to 4), which combine to create an effective batch size of 8, balancing memory usage and training stability. We set the num_train_epochs to 3, meaning the model will see the entire dataset three times. The learning_rate is set to 1e-4, controlling the update step size, and paired with a cosine learning rate scheduler (lr_scheduler_type) and 50 warmup_steps for a smooth start. For efficiency, we leverage mixed-precision training (fp16 or bf16) and the memory-efficient adamw_8bit optimizer. Finally, we start fine-tuning the Llama 3.1 8B model for approximately 3,000 steps.

Training Optimizations

We used Weights & Biases to keep an eye on the training. This let us see the `train/learning_rate` and `train/loss` as the model trained over its approximately 3,000 steps across 13 hours.

Looking at the `train/learning_rate` plot in Figure 3, we can see the schedule we set up. It starts with a quick increase during the first 50 `warmup_steps`, hitting our target `learning_rate` of 1e-4 (0.0001). After that, it follows the smooth downward curve we wanted from the `cosine` `lr_scheduler_type`. This approach lets the model adjust carefully at the start and then fine-tune more precisely as training ends.

Chrysler Crossfire Model Benchmark Results

Model Evaluation / Llama 3 Herd Comparison

We evaluated the performance of our fine-tuned Llama 3.1 8B model to determine if the fine-tuning process successfully learned the specialized knowledge about the Chrysler Crossfire. The evaluation involved comparing our model against the baseline Llama 3.1 Instruct models (8B, 70B, and 405B parameters) using a set of five benchmark questions covering common Crossfire-specific facts and procedures.

The benchmark questions targeted key information a Crossfire owner might ask:
  1. What is the replacement battery type for my Chrysler Crossfire? (Battery Type) [Ground Truth: H6]
  2. What is the stock wheel size for the front of my Chrysler Crossfire? (Front Wheel Size) [Ground Truth: 18 inches]
  3. What model of head light should I use as a replacement for my Chrysler Crossfire? (Headlight Model) [Ground Truth: H7]
  4. What is the stock wheel size for the rear of my Chrysler Crossfire? (Rear Wheel Size) [Ground Truth: 19 inches]
  5. How do I perform a throttle reset for my Chrysler Crossfire? (Throttle Reset Proc.) [Ground Truth: Key to position 2, pedal down 5s, key off, pedal up]
To judge each model's accuracy, we developed a dashboard to determine if each model's response accuracy describes the ground truth. The dashboard calls OpenRouter's API to engage Meta's Llama 3 Herd, including Llama 3.1 8B/70B/405B Instruct models. This process prompts the Chrysler Crossfire Model, along each Llama 3 model the questions above. We then used Meta’s new Llama 4 Maverick as a judge to determine each model’s accuracy. Llama 4 Maverick will either label each model’s response as ‘Correct’ or ‘Incorrect’ based on the provided groundtruth.

def call_openrouter_api(model_name, prompt, api_key, is_evaluation=False):
    """Calls the OpenRouter API for generation or evaluation."""
    log_prefix = "Evaluating with" if is_evaluation else "Querying"
    st.write(f"{log_prefix} OpenRouter: {model_name}...")

    messages = [{"role": "user", "content": prompt}]
    max_tokens = 50 if is_evaluation else 1024 # Shorter for eval expected output
    temperature = 0.1 if is_evaluation else 0.7 # Low temp for consistent eval

    try:
        response = requests.post(
            url="https://openrouter.ai/api/v1/chat/completions",
            headers={
                "Authorization": f"Bearer {api_key}",
                "Content-Type": "application/json",
            },
            data=json.dumps({
                "model": model_name,
                "messages": messages,
                "max_tokens": max_tokens,
                "temperature": temperature,
            }),
            timeout=180
        )
        response.raise_for_status()
        data = response.json()
        content = data.get("choices", [{}])[0].get("message", {}).get("content", "Error: Could not parse response.")
        return content.strip()
    except requests.exceptions.RequestException as e:
        st.error(f"Error calling OpenRouter model {model_name}: {e}")
        return f"Error: API call failed - {e}"
    except Exception as e:
        st.error(f"An unexpected error occurred with model {model_name}: {e}")
        return f"Error: Unexpected error - {e}"

def call_ollama_api(prompt):
    """Calls the local Ollama API."""
    st.write(f"Querying Ollama: {OLLAMA_MODEL_TAG}...")
    headers = {"Content-Type": "application/json"}
    payload = json.dumps({"model": OLLAMA_MODEL_TAG, "prompt": prompt, "stream": False})
    try:
        response = requests.post(url=OLLAMA_API_URL, headers=headers, data=payload, timeout=180)
        response.raise_for_status()
        # Handle potential non-JSON responses or different structures if Ollama version changes
        try:
            data = response.json()
            content = data.get("response", f"Error: 'response' key missing in Ollama JSON. Response: {response.text[:200]}")
        except json.JSONDecodeError:
             content = f"Error: Ollama returned non-JSON response: {response.text[:200]}"
        return content.strip()
    except requests.exceptions.ConnectionError as e:
        st.error(f"Error connecting to Ollama at {OLLAMA_API_URL}. Is Ollama running? Error: {e}")
        return f"Error: Connection failed - {e}"
    except requests.exceptions.RequestException as e:
        st.error(f"Error calling Ollama model {OLLAMA_MODEL_TAG}: {e}")
        return f"Error: API call failed - {e}"
    except Exception as e:
        st.error(f"An unexpected error occurred with Ollama model {OLLAMA_MODEL_TAG}: {e}")
        return f"Error: Unexpected error - {e}"

def evaluate_response_llm(original_prompt, candidate_response, reference_answer, api_key):
    """Evaluates a candidate response against a user-provided reference answer using an LLM judge."""

    # Check prerequisites
    if not api_key:
        st.error("OpenRouter API Key is required for LLM-based evaluation.")
        return "Error: API Key Missing"
    if not reference_answer:
        st.warning("No reference answer provided. Skipping LLM evaluation.")
        return "N/A (No Ref. Answer)"
    if candidate_response.startswith("Error:"): # Don't evaluate if generation failed
        return "N/A (Generation Error)"

    # --- !!! REVISED Evaluation Prompt for Stricter Judgement !!! ---
    # This prompt emphasizes accuracy, unambiguity, and penalizes misleading info.
    evaluation_prompt = f"""You are a meticulous and impartial judge evaluating an AI model's response based on a known correct reference answer.

Original User Question:
\"\"\"
{original_prompt}
\"\"\"

Reference Answer (Considered Ground Truth):
\"\"\"
{reference_answer}
\"\"\"

Candidate Response (Generated by AI model):
\"\"\"
{candidate_response}
\"\"\"

**Your Task:** Analyze the **Candidate Response** strictly against the **Reference Answer**. The goal is to determine if the candidate provides the *correct and essential information* from the reference, without adding misleading or incorrect details relevant to the core question.

**Evaluation Criteria:**
1.  **Accuracy:** Does the candidate accurately state the key information from the reference? (e.g., If the reference says 'H7', does the candidate correctly identify 'H7'?)
2.  **Clarity & Unambiguity:** Is the correct information presented clearly? Is it mixed with incorrect suggestions that make the answer confusing or potentially lead the user astray? (e.g., Listing 'H7' alongside incorrect bulb types like 'H11' or '9004' *as equally valid options for the primary headlight* makes the answer ambiguous and INCORRECT, even if 'H7' is present).
3.  **Completeness (Essential Info):** Does the candidate include the *most critical* piece of information from the reference needed to answer the user's core question? (e.g., Identifying the bulb type).
4.  **Focus:** Does the candidate directly address the core question, or does it primarily offer generic advice while being vague about the specific answer contained in the reference? A response that mentions the correct answer buried in generic advice and incorrect options is less valuable and potentially INCORRECT.

A response is **Correct** ONLY IF it accurately reflects the essential information from the reference answer clearly and without significant misleading additions or ambiguity regarding the core point.

A response is **Incorrect** IF:
- It fails to provide the essential information from the reference.
- It provides the essential information but mixes it ambiguously with incorrect information (like wrong bulb types suggested as equals).
- It is overly vague and evasive, prioritizing generic advice over the specific answer known from the reference.

Based on this **strict evaluation**, comparing the **Candidate Response** to the **Reference Answer**, is the candidate response correct?

Respond with only the single word 'Correct' or 'Incorrect'. Do not provide any explanation or justification.
"""

    )

        return "Eval Error"
